DDoS Botnet Attack Volume Prediction Project¶

In this project, we analyze a dataset of DDoS botnet attacks on IoT devices. The goal is to predict the attack volume—defined as the number of network flows per minute—using regression models. We will aggregate flow-level data, perform data exploration, pre-process the data, and experiment with different modeling techniques, including linear regression, feature engineering, and Ridge Regression. We'll also explore visualizations such as scatterplots, heatmaps, and a PCA (Principal Component Analysis) to better understand our features.

Introduction¶

This project explores the prediction of DDoS attack volume using regression techniques on a dataset collected from IoT device attacks. The dataset, ddos_dataset.csv, contains detailed flow-level records with variables such as:

  • stime: The start time of a network flow (Unix timestamp).
  • flgs_number: Numeric representation of flow state flags.
  • proto_number: Numeric representation of the protocol used in the network flow.
  • sport: The source port number.
  • pkSeqID: A unique identifier for each flow record.

My goal is to aggregate this data on a per-minute basis to compute the attack volume (i.e., the number of flows per minute) and then apply various regression models to predict this volume. This approach will help network administrators anticipate and mitigate potential DDoS attacks.

Regression is a method used to understand and predict how one variable (the target) changes based on other variables (the predictors). In our project, we use linear regression to predict the attack volume (which is the number of flows per minute) using features like flgs_number, proto_number, and sport.

The basic idea behind linear regression can be summed up by this formula:

y = β₀ + β₁x₁ + β₂x₂ + ... + βₙxₙ + ε

Here's what each part means:

y is the value we're trying to predict (in our case, the attack volume). x₁, x₂, ..., xₙ are the input features (for example, flgs_number, proto_number, and sport). β₀ is the intercept, or the value of y when all the x values are zero. β₁, β₂, ..., βₙ are the coefficients that show how much each feature influences y. ε is the error term, which captures the difference between our predictions and the actual observed values. The goal of linear regression is to find the best values for these coefficients so that our predicted values are as close as possible to the actual values. We often measure the performance of our model using Mean Squared Error (MSE), which calculates the average squared difference between predicted and real values. Because MSE is in squared units, we usually take the square root to get the Root Mean Squared Error (RMSE). RMSE gives us an idea of the average prediction error in the same units as our target variable.

In short, linear regression fits a straight line through our data points and tells us how each feature contributes to predicting the outcome.

Experiment 1: Pre-processing¶

After exploring the data, we need to clean and prepare it for modeling. In this step, we:

  • Convert the stime column (Unix timestamp) into a proper datetime format.
  • Convert sport into a numeric format to avoid mixed types.
  • Create a new column minute by flooring the datetime to the nearest minute.
  • Aggregate the data by minute, calculating the mean values for features (flgs_number, proto_number, sport) and counting the number of flows (using pkSeqID) to define our target variable attack_volume.

This pre-processing ensures our data is in a suitable format for building our regression model.

In [90]:
# Convert 'stime' to datetime (assuming it's a Unix timestamp)
df['stime'] = pd.to_datetime(df['stime'], unit='s')

# Convert 'sport' to numeric (in case it's not already)
df['sport'] = pd.to_numeric(df['sport'], errors='coerce')

# Create a new column 'minute' by flooring the time to the nearest minute
df['minute'] = df['stime'].dt.floor('T')

# Aggregate data by minute:
# - Compute mean of 'flgs_number', 'proto_number', 'sport'
# - Count flows (using 'pkSeqID') as attack_volume
aggregated = df.groupby('minute').agg({
    'flgs_number': 'mean',
    'proto_number': 'mean',
    'sport': 'mean',
    'pkSeqID': 'count'
}).rename(columns={'pkSeqID': 'attack_volume'}).reset_index()

Experiment 1: Baseline Linear Regression Model Summary¶

In this experiment, we developed a baseline model to predict DDoS attack volume (defined as the number of network flows per minute) using aggregated data. The key steps and insights from this experiment are summarized below:

Steps Involved:¶

  1. Data Aggregation and Pre-processing:

    • Conversion: We converted the stime column from a Unix timestamp to a datetime format.
    • Time Flooring: The datetime values were floored to the nearest minute to create a new column, minute.
    • Aggregation: We aggregated the flow-level data by minute, counting the number of flows (using pkSeqID) to create the target variable attack_volume. Additionally, we computed the mean values for important features like flgs_number, proto_number, and sport.

  2. Feature Selection & Data Splitting:

    • Feature Selection: We selected three key aggregated features: flgs_number, proto_number, and sport.
    • Data Splitting: The dataset was split into training (80%) and testing (20%) sets to allow for unbiased model evaluation.

  3. Model Building:

    • We built a baseline linear regression model using Scikit-learn's LinearRegression.
    • The model was trained on the training set and then used to predict the attack volume on the test set.

  4. Model Evaluation:

    • The performance of the model was evaluated using the Root Mean Squared Error (RMSE).
    • The baseline linear regression model achieved an RMSE of 80,456.24, which is relatively high.

Insights:¶

  • High RMSE: The high RMSE indicates that the baseline model struggles to capture the complex patterns in the data. This suggests that the selected features may not be sufficient or that the relationship between the features and the attack volume might be more complex than a simple linear model can capture.
  • Benchmark Establishment: This baseline performance serves as a benchmark for subsequent experiments. In further experiments, we explore additional features (such as time-based variables) and alternative modeling techniques (e.g., Ridge Regression) to improve prediction accuracy.
  • Next Steps: The findings from this experiment underscore the need for additional feature engineering and regularization to better model the intricacies of DDoS attack traffic.

Overall, Experiment 1 provided a starting point and established the groundwork for further model refinement and experimentation.

Impact¶

Accurate prediction of DDoS attack volume can have significant implications for network security:

  • Positive Impacts:

    • Proactive Defense: Improved predictions enable network administrators to allocate resources more efficiently and mitigate attacks before they cause damage.
    • Cost Efficiency: Accurate forecasting may help avoid over-provisioning resources during an attack.
    • Enhanced Network Resilience: Early warnings can minimize downtime and improve overall network stability.

  • Ethical Considerations and Risks:

    • Model Dependence: Over-reliance on model predictions might lead to false positives/negatives if the model doesn’t adapt to evolving attack strategies.
    • Data Bias: The training data might contain biases, which can propagate into the predictions.
    • Privacy: Handling network data requires stringent measures to ensure sensitive information remains protected.

Conclusion¶

This project explored the application of regression techniques to predict DDoS attack volume using aggregated flow-level data. Key takeaways include:

  • Data Aggregation: Converting raw network flows into a per-minute attack volume metric is crucial for meaningful predictions.
  • Baseline Modeling: A simple linear regression model provided a starting point, though its high RMSE highlighted the complexity of the task.
  • Feature Engineering: Incorporating time-based features offered insights into temporal patterns, even if the initial results did not improve RMSE.
  • Regularization: Ridge Regression reduced overfitting and improved performance compared to the baseline model.
  • Visual Insights: Scatterplots, heatmaps, and PCA helped visualize relationships between features and identify patterns in the data.

Overall, this iterative process underscored the importance of data pre-processing, feature selection, and model regularization in predictive analytics for network security.
